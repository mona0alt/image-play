Go 后端调用模型是否走 Higress？

推荐走 Higress，让 AI 插件生效

两种模式对比

模式 A：Go → Higress → 模型（推荐）
Go 后端
↓ 内部网络
Higress
限流 · 缓存 · 审计
↓ 公网
外部模型 API

✅ Higress 的 AI 插件对生成流量生效
✅ 统一限流，防止压垮模型
⚠️ 多一跳内部网络（Docker 内 1ms 以内，可忽略）

模式 B：Go → 直调模型
Go 后端
↓ 公网
外部模型 API

✅ 少一跳网络
❌ Higress 限流/缓存对生成无效
❌ 模型调用分散，难以统一管控

推荐：模式 A

既然已经部署了 Higress，核心目的就是统一管理模型调用。Go 后端调用模型时走 Higress（内部 Docker 网络地址，如 http://higress:8080/model/xxx），这样：

  • 限流生效：防止单个用户刷爆模型 API
  • 缓存生效：相同 prompt 可能命中缓存，节省成本
  • 审计生效：所有 prompt 和结果都被 Higress 记录
  • 模型切换：只需改 Higress 配置，Go 代码不动

内部 Docker 网络多一跳的延迟在 1ms 以内，完全可以忽略。

Higress 的双重角色

  • 公网入口：小程序 → Higress(443) → /api/* → Go 后端
  • 内部 AI 网关：Go 后端 → Higress(内部 8080) → /model/* → 外部模型
