为什么同时需要 Higress 和 Nginx？

其实可以省掉一个

原始设计的问题

你问得很对。Higress 本身就是基于 Envoy 的网关，已经具备反向代理、路由、HTTPS 能力。再叠一层 Nginx 是多余的。

调整后的方案：只用 Higress 做网关

简化后的架构
Higress（唯一入口）
80/443 → HTTPS 终止 → 路由分发
↓
/api/*
→ Go 后端
/admin/*
→ Go 后端 serve 静态文件
/model/*
→ 直调模型 API

具体调整

  • Higress：统一入口，处理 HTTPS、路由、限流、AI 插件
  • Go 后端：除业务 API 外，同时 serve /admin 路径下的管理后台静态文件（Vue3 build 产物）
  • 去掉 Nginx：静态文件由 Go 的 http.FileServer 提供，生产环境足够

好处

  • 少跑一个容器，节省约 20~30MB 内存
  • 配置更少，部署更简单
  • Higress 的 AI 插件（prompt 审计、内容安全）可以直接作用于所有流量
