# AI 图片小程序前端艺廊化重构设计

## 1. 文档目标

本设计只解决一件事：

- 在不改后端能力和核心业务对象的前提下，把当前前端从“功能可用但页面割裂的工具集合”重构为“统一、克制、接近参考稿的艺廊式微信小程序”

本轮重点不在新功能扩展，而在以下三件事：

- 统一五个页面的视觉语言和信息架构
- 重排主链路，让用户从“看作品”自然过渡到“开始创作”
- 保留现有数据流和接口，避免把前端换皮演变成全链路返工

---

## 2. 已确认约束

### 2.1 目标端

- 只面向微信小程序
- 不考虑 H5 适配

### 2.2 视觉来源

- 直接参考 `/Users/lili/Downloads/stitch_lumina_ai_image_studio`
- 保持参考稿的风格，不另做新风格探索

### 2.3 重构范围

- 覆盖 `首页 / 场景页 / 结果页 / 历史页 / 我的页`
- 允许重组页面模块、导航层级和交互顺序
- 不新增后端字段
- 不修改核心生成、支付、登录、历史接口的能力边界

### 2.4 产品目标

- 首页优先承担“作品馆”角色，而不是“立即生成”的工具台
- 底部导航调整为 `艺廊 / 创作 / 历史 / 我的`
- `创作` 页采用“先选场景，再选模板，再填写”的链路
- `我的` 页采用品牌化个人页，而不是工具菜单堆叠
- `结果` 页采用作品展示优先，而不是操作按钮优先

---

## 3. 当前问题

### 3.1 页面风格割裂

- `home / scene / history / profile / result` 目前各自成页，但排版、圆角、色彩、按钮和留白节奏不一致
- 大部分页面仍是“默认业务页”结构，无法形成参考稿那种策展式观感

### 3.2 信息架构偏工具化

- 首页只是在渲染 hero 和场景卡，缺少明确的策展层级
- 场景页像表单页，缺少“进入工作室”的过渡
- 历史页和我的页更像后台列表，不像作品产品

### 3.3 主链路缺少视觉主次

- 用户从首页进入生成流程后，页面很快退化成普通表单和普通列表
- 结果页虽然已经能区分状态，但仍是“图片 + 按钮”的简单容器，无法承接作品展示的品牌感

### 3.4 组件边界能复用但表现层过弱

当前已有组件：

- `SceneHeroCard`
- `SceneGalleryCard`
- `TemplatePicker`
- `SceneFieldForm`
- `ResultPreviewCard`

这些组件承担的业务语义仍然有效，但当前表现层过于简陋，不足以支撑整站重构目标。

---

## 4. 设计结论

### 4.1 采用“艺廊直译型”重构

本轮采用最接近参考稿的方案：

- 首页承担策展和种草
- 创作页承担明确的开始创作入口
- 历史页承担作品档案
- 我的页承担品牌化账户与权益展示
- 结果页承担作品展示与后续动作

不采用“工作台优先”方案，也不把首页做成表单聚合页。

### 4.2 保持数据边界不变

前端继续复用当前数据边界：

- `GET /api/configs/client` 提供 `scene_order`
- `GET /api/scenes/:scene_key/templates` 提供模板和表单 schema
- `POST /api/generations` 提交生成
- `GET /api/history` 提供结果页、历史页和我的页最近作品所需数据

本轮不新增前端必须依赖的新接口。

### 4.3 以页面层重组为核心

本轮主要改动在：

- 页面层布局
- 页面之间的导航关系
- 组件的视觉层和分组方式
- 状态页、空状态、错误态的表达方式

而不是重写 store、services 或重新定义领域对象。

## 5. 信息架构

### 5.1 底部导航

底部导航统一为：

- `艺廊`
- `创作`
- `历史`
- `我的`

其中：

- `艺廊` 是默认首页
- `创作` 是独立主入口
- `历史` 是作品档案页
- `我的` 是品牌化个人与权益页

`结果页` 不进入底部导航，作为详情页由创作链路和历史链路共同进入。

### 5.2 页面角色分工

#### 艺廊

- 展示主推场景和场景艺廊
- 承担“被吸引、被说服、被带入”的角色
- 保留少量最近生成和权益提示，但不抢主视觉

#### 创作

- 承担“明确开始创作”的角色
- 先选场景，再选模板，再填表单
- 不做全功能工作台

#### 历史

- 承担“作品档案与状态记录”的角色
- 展示作品缩略图、状态、时间和进入结果页的路径

#### 我的

- 承担“个人藏馆 + 账户权益”的角色
- 展示余额、免费额度、套餐、最近作品和常用场景

#### 结果

- 承担“作品展示”的角色
- 把操作按钮放在作品之后，而不是作品之前

### 5.3 主链路

主链路重组为：

1. 用户进入 `艺廊`
2. 在 hero 或场景卡上选择感兴趣的场景
3. 进入 `创作` 对应场景
4. 选择模板并填写参数
5. 提交生成并跳到 `结果`
6. 生成完成后沉淀到 `历史` 与 `我的-最近作品`

这条链路强调“先看作品，再做作品”，而不是“先看到表单，再理解产品”。

## 6. 页面设计

### 6.1 艺廊页

#### 目标

- 承担默认首页
- 用最少的信息建立产品气质与可创作场景

#### 结构

1. 顶部品牌栏
   - 轻量品牌标题
   - 右侧个人头像或账户入口
2. Hero 场景区
   - 一个主推场景大卡
   - 强调图片、标题、描述和主 CTA
3. 场景艺廊区
   - 2 列场景卡
   - 保持作品纵向比例
4. 最近生成区
   - 横向缩略图带
   - 展示用户最近生成的少量作品
5. 轻量权益区
   - 展示剩余额度或购买入口
   - 只做提醒，不做首页主角

#### 交互

- 点击 hero 或场景卡进入对应 `创作` 场景
- 点击最近作品进入对应 `结果`
- 点击权益卡进入 `我的`

### 6.2 创作页

#### 目标

- 承接从艺廊进入后的创作意图
- 形成“场景 -> 模板 -> 表单 -> 提交”的线性工作流

#### 结构

1. 场景头图区
   - 当前场景封面
   - 场景标题
   - 一句场景说明
2. 模板选择区
   - 模板卡采用封面化卡片
   - 展示模板标题、风格标签、选中态
3. 参数填写区
   - 使用服务端 `form_schema`
   - 表单顺序按 schema 渲染
4. 底部提交区
   - 主按钮固定在底部安全区上方
   - 保持单一明确动作

#### 交互

- 页面进入时自动按 `scene_key` 加载模板
- 默认选中第一个模板
- 切换模板时保留统一的页面壳，不做页面跳变
- 提交前继续沿用必填校验
- 提交成功后进入 `结果`

### 6.3 历史页

#### 目标

- 从“列表页”升级为“作品档案页”

#### 结构

1. 顶部标题区
   - 简短标题
   - 说明文案可选
2. 状态筛选区
   - 胶囊筛选条
   - 保持一行横向滚动或轻量换行
3. 作品档案列表
   - 以缩略图为主
   - 辅以状态、时间和模板/场景信息

#### 交互

- 点击任一记录进入 `结果`
- `success` 展示作品缩略图
- `queued / running / result_auditing` 展示占位图和状态文案
- `failed` 展示失败态卡片，但视觉弱于成功态

### 6.4 我的页

#### 目标

- 从“功能集合页”升级为“品牌化个人页”

#### 结构

1. 账户头部
   - 头像
   - 账户标题
   - 一句简短品牌说明
2. 权益卡
   - 余额
   - 免费额度
   - 当前权益说明
3. 最近作品区
   - 横向作品带
4. 常用场景区
   - 快捷返回高频场景
5. 套餐购买区
   - 展示套餐卡和购买按钮

#### 交互

- 点击最近作品进入 `结果`
- 点击常用场景进入 `创作`
- 点击套餐卡触发当前下单逻辑

### 6.5 结果页

#### 目标

- 把结果页重做为作品展示页

#### 结构

1. 大图展示区
   - 结果图绝对优先
2. 摘要区
   - `prompt` 摘要
   - 风格标签
3. 操作区
   - 主按钮：保存到相册
   - 次按钮：微信分享
4. 延伸动作区
   - 再来一张
   - 相关推荐作品或最近作品

#### 状态分支

- `success`
  - 展示作品、摘要和主操作
- `queued / running / result_auditing`
  - 展示大图占位、状态说明和等待中的作品页壳
- `failed`
  - 展示失败说明和重新生成入口
- `not found`
  - 展示轻量错误卡和返回创作入口

## 7. 视觉系统

### 7.1 风格原则

直接继承参考稿的视觉语言：

- 暖白底
- 深色文字
- 极浅灰分层
- 细边框
- 弱阴影
- 大留白
- 策展式模块节奏

不引入高饱和品牌色，不走“互联网功能页”视觉。

### 7.2 排版

- 标题层级清晰，一级标题保持强识别
- 小标签统一使用轻量 uppercase/tracking 风格
- 正文克制，避免过长说明打断作品感

### 7.3 卡片系统

卡片分为两类：

- 大型展示卡
  - 用于 hero、作品展示、重点内容
  - 圆角更柔和
- 功能卡与胶囊
  - 用于筛选、权益、按钮辅助模块
  - 圆角更收敛

卡片主要依靠底色、边框和留白建立层级，避免使用重阴影。

### 7.4 图片策略

- 首页场景卡、创作模板卡、历史卡、我的最近作品统一优先使用纵向作品比例
- 图片不作为装饰，而作为结构主角
- 没有真实图片时，使用风格统一的占位骨架，而不是默认灰块

### 7.5 按钮系统

- 主按钮：黑底白字
- 次按钮：白底黑边
- 结果页保持一主一辅双按钮结构
- 所有主操作按钮贴近拇指热区，同时与底部导航保持层级分离

### 7.6 底部导航

- 图标与文案风格克制
- 视觉上接近参考稿的轻量导航
- 不做夸张激活态和高对比彩色高亮

## 8. 组件重构策略

### 8.1 可保留的业务语义

以下组件的业务职责可以保留：

- `SceneHeroCard`
- `SceneGalleryCard`
- `TemplatePicker`
- `SceneFieldForm`
- `ResultPreviewCard`

### 8.2 需要重做的表现层

#### SceneHeroCard

- 从“图标 + 名称 + 描述”升级为“策展 hero 卡”
- 加入图片主视觉、标签、标题、描述和 CTA

#### SceneGalleryCard

- 从简单功能格子升级为作品型场景卡
- 支持封面、标题、简短副标题

#### TemplatePicker

- 从纯文字模板块升级为模板封面卡组
- 支持封面图、风格标签、选中态和更强的纵向节奏

#### SceneFieldForm

- 继续使用动态 schema
- 输入框、选择器、文本域统一为轻边框和弱背景风格
- 强化表单的节奏感和可读性

#### ResultPreviewCard

- 从“图片 + 两个按钮”的简单容器升级为结果页主视觉容器
- 承载图片展示、摘要、标签和操作分组

### 8.3 新增页面壳层

建议新增轻量页面壳或样式层，用于统一：

- 页面顶部留白
- 分区标题样式
- 卡片间距
- 安全区底部 padding
- 固定底部 CTA 的页面布局

## 9. 状态与异常设计

### 9.1 数据流

- `艺廊` 读取 `scene_order` 组织主推场景和艺廊顺序
- `创作` 读取当前场景模板并默认选中首个模板
- 提交生成后跳转 `结果`
- `结果` 通过历史数据和 `generation_id` 找到当前记录
- `历史` 和 `我的` 复用相同历史数据源做不同表达

### 9.2 加载态

加载态统一使用页面壳内的占位表达：

- `艺廊` 使用 hero 骨架和场景卡骨架
- `创作` 使用封面骨架、模板骨架和表单骨架
- `结果` 使用作品占位与状态文案
- `历史 / 我的` 使用缩略图骨架和卡片骨架

不再只显示“加载中...”文本。

### 9.3 空状态

- `历史` 为空时，展示引导去创作
- `我的` 没有最近作品时，展示常用场景入口
- `结果` 找不到记录时，展示轻量错误卡并给出返回创作入口

### 9.4 错误态

- 首页配置加载失败时，保留页面壳并提供重试入口
- 创作提交失败时，必须保留当前选择和已填写内容
- 历史加载失败时，页面内展示重试容器，不只依赖 toast
- 结果失败态展示失败文案与“重新生成”入口

## 10. 测试与验证

### 10.1 测试目标

确认本轮重构没有误伤现有业务逻辑，同时覆盖新的页面分支和流程重组。

### 10.2 自动化验证

至少覆盖以下验证：

1. 现有 `store / utils / services` 单测继续通过
2. 新增页面层最小必要单测，重点覆盖：
   - 按场景进入创作页
   - 模板默认选中
   - 必填项校验
   - 结果页状态分支展示
3. 运行：
   - `npm test`
   - `npm run type-check`

### 10.3 人工走查

在微信小程序端做一次完整走查，重点确认：

- 底部导航切换是否自然
- 页面滚动节奏是否接近参考稿
- 固定 CTA 与安全区是否冲突
- 结果页主次是否正确
- 空状态、加载态和错误态是否仍保持统一风格

## 11. 非目标

本轮明确不做以下内容：

- H5/Web 适配
- 新的后端接口与字段扩展
- 账户体系重构
- 充值逻辑重写
- 新的生成能力
- 模板编辑器
- 社区或作品分享广场

## 12. 实施边界

### 12.1 允许改动

- `frontend/src/pages/*`
- `frontend/src/components/*`
- 必要的页面样式与公共壳层
- 与页面重构直接相关的最小测试补充

### 12.2 尽量不动

- `frontend/src/services/*`
- `frontend/src/store/*`
- `frontend/src/types/*`

除非页面重构需要补一个很小的适配层，否则不应借机改动业务边界。

### 12.3 成功标准

如果以下条件同时成立，则认为本轮重构达成目标：

- 五个页面在视觉和导航上形成一个统一产品
- 首页首先像作品馆，而不是工具页
- 创作链路比当前更清晰
- 结果页具有作品展示感
- 历史页和我的页摆脱后台列表感
- 自动化验证通过，且小程序人工走查无明显回退
