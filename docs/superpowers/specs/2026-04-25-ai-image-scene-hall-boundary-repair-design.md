# AI 图片场景馆边界修复设计

## 1. 文档目标

本设计只解决一件事：

- 把当前项目从“文档上已完成、代码里仍有 mock 和断链”修正为“主业务链真实可用”

本轮目标不是做完所有上线能力，而是先把最核心的 `登录 -> 选场景 -> 选模板 -> 提交生成 -> 查看历史/结果` 闭环打通，并收紧关键边界，避免继续在错误抽象上堆代码。

---

## 2. 当前问题

当前实现存在四类结构性问题：

### 2.1 前端主链路未真正接后端

- 场景页仍使用本地 `mockTemplates`
- 生成提交仍是 `console.log`
- 结果页依赖 URL 直接传 `result_url`
- 401 跳转到不存在的登录页

### 2.2 后端用户边界是假的

- 登录接口返回固定用户 ID
- 用户未落库
- `/me`、下单、生成等依赖真实用户数据的接口无法稳定工作

### 2.3 模板驱动链路没有打通

- 数据库有 `scene_templates`
- 后台可以开关模板
- 前端和 worker 却没有真正消费模板数据

### 2.4 状态与统计口径不一致

- 生成状态存在多套定义
- worker、历史页、后台统计使用的状态值不一致

这些问题如果不先修，继续实现支付和后台权限只会扩大返工面。

---

## 3. 本轮范围

### 3.1 包含

- mock 登录改为真实用户查找/创建
- `/me` 基于真实用户返回数据
- 首页客户端配置与真实五个场景一致
- 模板由后端真实下发
- 生成任务由前端真实提交
- worker 复用统一 Prompt Builder
- 历史页和结果页基于真实任务数据工作
- 统一生成状态枚举
- 补齐覆盖上述行为的后端和前端测试

### 3.2 不包含

- 真实微信登录
- 真实微信支付回调验签
- 真实管理员角色体系
- COS 真实签名上传
- 计费从“成功后扣费”重构为“预占或前置冻结”

### 3.3 Breaking Changes

本轮允许做以下 breaking 调整：

- 统一生成状态为 `queued / running / result_auditing / success / failed`
- 调整客户端配置返回内容，使其与真实场景馆一致
- 新增模板查询接口
- 前端模板、生成、结果页数据流按新接口重组

---

## 4. 设计结论

### 4.1 本轮采用“最小闭环重构”

项目继续沿用当前 `UniApp + Vue3 + Go API + Go Worker + PostgreSQL` 骨架，但要收紧三条明确边界：

1. `auth/user`
   登录只负责把外部登录凭证映射成系统内用户并签发 token。
2. `template/config`
   场景配置和模板是前后端的单一真源，不再允许页面本地 hardcode 业务模板。
3. `generation/history`
   生成记录是结果页和历史页的唯一真源，页面不再依赖 URL 拼接完整业务数据。

### 4.2 为什么不做“大重构”

当前问题的根因不是技术栈，而是边界失真：

- 真数据和 mock 数据并存
- 表结构存在但运行链路未消费
- 状态没有单一真源

因此本轮以“打通真实闭环”为目标，而不是把整个后端一次性重写成完整领域架构。

## 5. 后端设计

### 5.1 用户与认证边界

#### 目标

让用户成为真实持久化实体，所有需要用户上下文的接口都基于数据库中的用户记录工作。

#### 方案

- `POST /api/auth/login`
  - 继续接受 mock `code`
  - 将 `code` 映射为 `openid = mock-openid-<code>`
  - 按 `openid` 查找用户
  - 若不存在则创建新用户，默认 `balance = 0`、`free_quota = 3`
  - 返回真实用户数据和 JWT
- `GET /api/me`
  - 直接查询真实用户
  - 返回余额和免费额度

#### 边界约束

- 不允许再返回固定用户 ID
- 不允许再出现“token 合法但数据库中没有用户记录”的正常路径

### 5.2 客户端配置边界

#### 目标

首页和场景页只能消费与产品规划一致的真实配置。

#### 方案

- `GET /api/configs/client` 返回：
  - `brand_slogan`
  - `scene_order = ["portrait", "festival", "invitation", "tshirt", "poster"]`
  - `pricing`
- 本轮不把模板塞进客户端配置
- 客户端配置只负责首页入口排序和基础文案，不承载模板 schema

#### 设计原因

模板是按场景加载的数据，和首页基础配置不是同一层级。把两者分开可以降低首页加载负担，也能避免场景页继续依赖静态 mock。

### 5.3 模板边界

#### 目标

数据库中的 `scene_templates` 必须成为模板列表和表单 schema 的单一真源。

#### 方案

- 新增接口：`GET /api/scenes/:scene_key/templates`
- 查询条件：
  - `scene_key = ?`
  - `is_active = true`
- 返回字段：
  - `template_key`
  - `name`
  - `form_schema`
  - `sample_image_url`

#### 不做的事

- 本轮不做后台模板编辑器
- 本轮不把模板开关做成前端后台页面

### 5.4 生成边界

#### 目标

任务创建时必须验证模板是否合法，任务执行时必须走统一 Prompt Builder。

#### 方案

- `POST /api/generations`
  - 校验 `scene_key` 是否受支持
  - 校验 `template_key` 是否属于该场景且处于启用状态
  - 校验通过后再创建任务
  - 初始状态固定写入 `queued`
- worker 执行任务时：
  - 不再使用自定义字符串拼接逻辑
  - 改为调用 `scenes.BuildPrompt`
  - 生成成功后统一写 `success`
  - 审核或模型失败统一写 `failed`

#### 设计原因

模板校验必须前置到创建阶段，否则数据库中会积累大量“模板根本不存在”的脏任务；Prompt Builder 也必须只有一套，否则模板数据和生成结果会持续偏离。

### 5.5 历史与结果边界

#### 目标

历史页和结果页都以 `generations` 中的真实记录为准。

#### 方案

- 保留 `GET /api/history`
- 返回字段继续包含：
  - `id`
  - `scene_key`
  - `template_key`
  - `status`
  - `result_url`
  - `created_at`
- 本轮不新增独立详情接口，结果页先通过已有历史数据和 `generation_id` 恢复结果

#### 约束

- 页面不再依赖 URL 里传完整业务状态
- 页面最多传 `generation_id`

### 5.6 状态真源

本轮统一状态枚举：

- `queued`
- `running`
- `result_auditing`
- `success`
- `failed`

所有以下位置必须统一：

- migration 约束
- repository 查询
- worker 状态推进
- 历史页筛选
- 后台指标统计

## 6. 前端设计

### 6.1 全局数据流

#### 首页

- 启动时请求 `GET /api/configs/client`
- 只消费 `scene_order` 和品牌文案
- 渲染逻辑修正为：
  - 有场景时显示内容
  - 无场景且无错误时显示加载态
  - 有错误时显示错误态

#### 场景页

- 根据 `scene_key` 请求 `GET /api/scenes/:scene_key/templates`
- 使用服务端返回的 `form_schema` 驱动表单
- 取消本地 `mockTemplates`

#### 生成提交

- `generation store` 新增真实提交能力
- 提交时组装：
  - `client_request_id`
  - `scene_key`
  - `template_key`
  - `fields`
- 成功后保存最近一次 `generation_id`

#### 结果页

- 页面只接收 `generation_id`
- 优先从已有历史数据中匹配任务
- 若没有匹配结果，则显示空态或引导返回历史页

#### 历史页

- 过滤值使用统一状态
- 不再使用和后端不一致的 `success/failed/all` 之外的隐式状态映射

### 6.2 Store 设计

#### `config store`

只保留客户端配置，不承载模板或生成状态。

#### `generation store`

负责：

- 当前场景
- 当前模板列表
- 选中模板
- 表单值
- 提交中状态
- 最近一次 `generation_id`
- 最近一次提交错误

不负责：

- 用户信息
- 首页配置

#### `user store`

继续负责 token 与用户信息，但 401 清理 token 后的跳转要改为当前存在的页面。

### 6.3 401 行为

本轮不新增登录页。

统一策略：

- 清理本地 token
- 回到已存在的页面入口，例如首页
- 由后续真实登录页或微信静默登录再补全

## 7. 数据与接口约定

### 7.1 模板返回结构

前端模板类型统一为：

- `key`
- `name`
- `sceneKey`
- `formSchema`
- `sampleImageUrl`

后端接口字段可以是 snake_case，但前端 API 层要做一次映射，页面内部不直接依赖后端原始字段。

### 7.2 生成请求

生成请求固定包含：

- `client_request_id`
- `scene_key`
- `template_key`
- `fields`

本轮暂不把高级 prompt 透传进后端主链路。高级设置 UI 可以保留，但不能继续伪装成已经影响生成结果。

因此本轮选择：

- 若高级 prompt 尚未接入后端，则前端先隐藏或禁用该输入
- 不允许继续展示“看起来可用、实际上无效”的字段

### 7.3 客户端配置

客户端配置的职责被限定为：

- 品牌文案
- 场景排序
- 基础价格展示

不包含：

- 模板列表
- 表单 schema
- 任务状态

## 8. 测试设计

### 8.1 后端

先写失败测试，再改实现。

本轮至少补齐以下测试：

- `auth_handler`
  - 首次登录时自动创建用户
  - 重复登录同一 code 复用已有用户
- `templates handler/service/repo`
  - 只返回 `is_active = true` 的模板
  - 场景不匹配时返回空数组
- `generations handler/service`
  - 禁用模板不能创建任务
  - 合法模板创建后状态为 `queued`
- `generation job`
  - 执行任务时使用 `scenes.BuildPrompt`
  - 成功后状态为 `success`
- `admin metrics`
  - 成功统计口径改为 `success`

### 8.2 前端

本轮至少补齐以下测试：

- `generation store`
  - 加载模板列表
  - 模板选择与表单更新
  - 提交流程中的 loading、成功、失败状态
- `api`
  - 模板接口与生成接口的响应映射
- 纯逻辑测试
  - 首页内容/加载/错误态分支
  - 历史过滤基于统一状态工作

1. `cd backend && go test ./...`

## 9. 风险与后续

### 9.1 本轮接受的已知风险

- 支付仍是 mock，支付安全问题未在本轮解决
- 管理后台仍未建立真正的角色权限体系
- 结果页没有独立详情接口，当前依赖历史数据恢复结果

### 9.2 下一轮推荐顺序

1. 支付回调与订单模型修复
2. 管理后台鉴权与角色边界
3. 生成扣费模型从“成功后补扣”升级为“前置校验 + 幂等结算”

---

## 10. 成功标准

本轮完成后，应满足以下条件：

- 任意 mock `code` 登录后都能落到真实用户记录
- 首页只展示真实五个场景，不再出现错误场景键
- 场景页模板全部来自后端接口，不再依赖本地 mock 数据
- 提交生成会真实创建任务记录
- worker 成功处理后，历史页能看到 `success` 结果
- 结果页不再依赖 URL 中硬编码的完整结果数据
- 代码中不存在“模板表存在但运行链路不消费它”的双轨实现
