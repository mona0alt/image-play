# 艺廊首页入口叠册卡组设计文档

## 1. 背景

当前艺廊首页的入口区由三部分组成：

- 一张 `SceneHeroCard`
- 一张单独的「面相分析」工具卡
- 一组两列 `SceneGalleryCard`

这套结构的问题不是功能不全，而是视觉语言分裂：

- `人像写真` 主卡和其他入口卡不在同一个系统里
- `面相分析` 像工具快捷入口，不像艺廊内容的一部分
- 两列宫格缺少陈列感，难以做出清晰且有质感的入口层级

本次改造只解决首页入口区的统一，不扩展功能，不重做其他页面。

---

## 2. 目标

本次设计目标如下：

- 把首页入口区改成单一的叠册式卡组，而不是 `主卡 + 工具卡 + 宫格` 的拼接结构
- 将 `面相分析` 并入同一组入口卡片，和 `人像写真 / 节日海报 / 邀请函 / T恤图案 / 商业海报` 使用同一视觉系统
- 保留清晰度优先的前提下做出更强的高级质感和光泽感
- 让用户在卡组区域内上滑时，能依次翻看下一张卡片，而不是立即离开入口区

成功标准：

- 用户一眼能看出入口区是一套统一的作品卡系统
- 每张卡片的边缘轮廓清晰，露边卡不会糊成一团
- 卡面看起来有高端卡材质感，但不会像真实银行卡界面
- 首页现有的数据和跳转链路保持稳定

---

## 3. 范围与非目标

### 3.1 本次范围

- 仅调整 `frontend/src/pages/home/index.vue` 的入口区结构和视觉呈现
- 调整首页 view-model，使 `面相分析` 作为展示卡插入首页入口序列
- 新增或替换首页入口区组件，用于承载叠册卡组和滑动切换行为
- 保持最近作品区、额度卡等首页其他模块继续存在

### 3.2 非目标

- 不改 `scene_order` 后端返回结构
- 不改创作页 `/pages/scene/index`
- 不改 `face-reading` 页面内部 UI
- 不改结果页、历史页、我的页
- 不新增后端接口或字段

---

## 4. 当前代码上下文

当前首页主要由以下部分构成：

- `frontend/src/pages/home/index.vue`
  - 渲染 `SceneHeroCard`
  - 渲染单独的「面相分析」卡
  - 渲染 `SceneGalleryCard` 两列列表
- `frontend/src/pages/home/view-model.ts`
  - 提供 `heroScene`、`galleryScenes`、`recentWorks` 和额度信息
- `frontend/src/utils/scene.ts`
  - 维护场景展示元数据和默认顺序解析

已有事实：

- 创作场景顺序来自 `GET /api/configs/client` 返回的 `scene_order`
- `面相分析` 不是场景数据的一部分，而是独立入口
- 首页最近作品和额度卡已经可用，不需要为本轮改造返工

因此，本次改造的核心是重组首页的展示模型和入口组件，而不是改业务数据边界。

---

## 5. 设计结论

### 5.1 采用单一“竖向叠册式卡组”

首页入口区改为一个固定高度的叠册卡组：

- 当前激活卡完整展示
- 后续卡片在下方整齐对齐叠放
- 每张后卡只露出一段顶部边缘
- 用户在卡组区域内向上滑时，按张切换到下一张卡

不采用明显错位压叠方案，原因是用户已明确要求：

- 卡片边缘需要整齐对齐
- 清晰度和高级感优先于夸张的错位效果

### 5.2 `面相分析` 并入同一入口系统

首页入口序列统一为一组展示卡，当前顺序按现有产品语义处理：

1. 第一张仍由 `scene_order` 的首个场景承担主入口
2. `面相分析` 插入到首页展示序列中，作为同组卡片的一员
3. 其余场景继续按 `scene_order` 依次展示

当前常见顺序示意：

- `人像写真`
- `面相分析`
- `节日海报`
- `邀请函`
- `T恤图案`
- `商业海报`

这里的“插入”只发生在首页展示模型，不回写到 `scene_order`，避免把工具入口和场景配置耦合。

### 5.3 卡片质感采用“高端银行卡材质感”

视觉方向借用高端银行卡的材质感，但不借用其信息布局。

保留元素：

- 深浅渐变底
- 柔和镜面高光
- 细金属边线
- 轻微压印/封章感
- 干净、锐利的卡片边缘

明确不做：

- 卡号、有效期、发卡组织等真实金融界面符号
- 复杂金箔纹理
- 大面积流动反光动画
- 低清晰度的模糊大光晕

## 6. 页面结构

### 6.1 首页入口区的新结构

首页内容顺序调整为：

1. 顶部页面壳和品牌信息
2. 叠册卡组入口区
3. 最近作品区
4. 额度卡

入口区不再拆成 hero、工具卡、宫格三个区块，而是只保留一个入口卡组容器。

### 6.2 卡组容器

卡组容器具备以下特征：

- 固定视觉高度
- 内部承载一张完整激活卡和若干露边卡
- 所有卡片宽度、圆角、描边和边缘位置完全一致
- 露边高度保持统一，形成稳定的节奏

### 6.3 卡片层级

卡片分两种密度：

#### 激活卡

- 展示完整标题
- 展示一句简短描述
- 展示 2 到 3 个标签
- 展示更完整的材质细节和光泽感

#### 露边卡

- 只展示眉标和标题
- 不展示长描述
- 保留同材质外观，但内容压缩到单层

这样可以同时满足：

- 当前卡有足够表现力
- 底卡数量增加时仍然清晰

---

## 7. 视觉系统

### 7.1 统一风格语言

所有入口卡都沿用 `人像写真` 主卡所代表的统一语言：

- 大圆角
- 深浅对照明显的高质感渐变
- 少量英文 eyebrow
- 克制标签
- 精致阴影
- 清晰描边

`面相分析` 不再使用“工具入口”式排版，而改为同等地位的主题卡。

### 7.2 材质层设计

卡面材质应体现“高级卡面”而不是“普通白卡片”：

- 第一层：深色或浅色主渐变
- 第二层：斜向高光涂层
- 第三层：极细金属边线或内描边
- 第四层：轻度局部压印/圆形封章装饰

这些效果都应通过清晰的 CSS 渐变、边框和阴影完成，尽量避免依赖位图纹理。

### 7.3 清晰度规则

为满足“边缘要清晰、分辨率高”的要求，视觉上遵守以下规则：

- 卡片轮廓必须有明确的边界线
- 卡片阴影不能发灰发糊，扩散范围要受控
- 不使用大面积模糊发光来营造高级感
- 露边卡的内容不超过一层主信息
- 标签数量控制在 2 到 3 个以内

### 7.4 配色策略

每张卡可以有独立配色，但必须属于同一材质体系：

- `人像写真`：深灰棕 / 炭黑暖调
- `面相分析`：深褐 / 炭黑金属调
- `节日海报`：暖铜 / 节庆暖光调
- `邀请函`：珠光米白 / 香槟纸面调
- `T恤图案`：黑灰蓝 / 冷感印刷调
- `商业海报`：烟灰 / 商业编辑调

色相可以不同，但明度控制、边框质感和高光方式必须一致。

## 8. 交互设计

### 8.1 滑动行为

用户在叠册卡组区域向上滑动时：

1. 先由卡组消费手势
2. 当前激活卡离场
3. 下一张卡补位成为新的激活卡
4. 后续卡继续保持露边排列

切换应是“按张吸附”的，而不是自由滚动。

### 8.2 页面滚动边界

为了避免与整页滚动冲突，交互边界定义如下：

- 在卡组未切到最后一张前，优先切卡
- 到最后一张后再继续上滑，整页恢复正常下滚
- 向下滑时，允许用户回看上一张卡
- 滑动距离不足阈值时，不切卡，直接回弹

### 8.3 点击行为

为了让交互更稳，卡组同时支持点击：

- 点击当前激活卡：进入对应页面
- 点击露边卡：该卡上浮为激活卡，再次点击进入

这样即使滑动行为在小程序设备上存在差异，入口操作仍然稳定可达。

### 8.4 跳转规则

- 场景卡点击后进入 `/pages/scene/index?scene_key=...`
- `面相分析` 卡点击后进入 `/pages/face-reading/index`

所有卡片的视觉样式统一，但跳转目标按现有能力边界区分。

---

## 9. 数据与组件边界

### 9.1 展示模型

首页需要从当前“hero + gallery”模型，收敛为统一入口卡序列模型。

建议在首页 view-model 中生成一个新的入口数组，每个元素至少包含：

- `kind`：`scene` 或 `tool`
- `key`
- `title`
- `description`
- `eyebrow`
- `tags`
- `accent`
- `path` 或等价跳转信息

其中：

- 场景卡继续基于 `buildScenePresentation`
- `面相分析` 卡使用本地静态元数据构建

### 9.2 组件边界

建议新增一个专用入口组件，例如：

- `GalleryEntryStack`

组件职责：

- 接收统一入口数组
- 管理当前激活索引
- 渲染完整卡与露边卡
- 处理滑动切换和点击切换

这样可以避免继续把逻辑散落在：

- `SceneHeroCard`
- `SceneGalleryCard`
- 首页页面模板

### 9.3 保留的数据边界

本轮继续复用：

- `scene_order`
- 用户资料数据
- 最近作品数据

不新增 store，不改后端 DTO，不新增接口。

---

## 10. 错误与状态处理

首页入口区虽然主要是静态展示，但仍需处理状态表达：

- 配置加载失败时：首页沿用现有错误卡，不展示半残入口区
- `scene_order` 为空时：退回到当前静态默认顺序 `portrait -> festival -> invitation -> tshirt -> poster`
- `面相分析` 卡不依赖额外接口，因此不引入新失败模式

卡组本身不需要 skeleton 动效升级为复杂版本，本轮重点仍是稳定和清晰。

---

## 11. 测试与验证

### 11.1 单元测试

需要覆盖：

- 首页入口数组生成顺序正确
- `面相分析` 正确插入展示序列
- 场景卡与工具卡跳转目标正确

### 11.2 组件测试

需要覆盖：

- 激活卡与露边卡状态切换正确
- 点击露边卡后能切换为激活卡
- 点击激活卡时触发正确事件

### 11.3 手动验证

必须在微信小程序环境中验证：

- 卡组内滑动与整页滚动是否冲突
- 快速滑动时是否跳卡过多
- 露边卡边缘在高密屏下是否足够清晰
- 深色卡和浅色卡上的文字对比是否足够

---

## 12. 最终结论

本次首页入口区重构采用以下最终方向：

- 结构上：由 `主卡 + 工具卡 + 宫格` 改为单一叠册卡组
- 内容上：`面相分析` 并入同一入口系统，但不写回 `scene_order`
- 视觉上：统一使用接近 `人像写真` 主卡的高端卡面语言
- 材质上：采用银行卡级漆面光泽质感，但不模仿真实银行卡信息布局
- 交互上：支持在卡组内按张滑动翻看，并与整页滚动做明确边界控制

该方案能在不改后端能力和核心生成链路的前提下，显著提升首页入口区的统一性、清晰度和高级质感。
